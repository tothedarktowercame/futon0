## 2. The narrow R12-unblocking subset

R12 says: a second AIF loop runs on a slower cadence and treats the inner loop's hyperparameters (preference priors, learning rates, EFE weights) as hidden state to be inferred.

The WM contract names the **smallest honest take-up path** at `~/code/futon2/docs/futon-aif-completeness.md:285`:

> Intrinsic-value learning over capability-gap recommendations: observe which emitted
> `:learn-action-class` recommendations the operator actually follows through on after R10
> is running long enough to accumulate evidence, then learn per-target-class intrinsic-value
> credit from acted-on vs ignored history.

**The R12-unblocking subset of Q6 is therefore:**

1. **A durable, queryable record of operator decisions on writer-action proposals.**
   This is the *input* the outer loop reads. Already partially present:
   - claude-2's consent-gate (`~/code/futon4/dev/arxana-vsatarcs-consent.el`) dispatches `:confirm` / `:reject` / `:ignore` / `:abstain-for-now` on each proposal.
   - VSATARCs trace (`~/code/futon4/dev/arxana-vsatarcs-trace.el`) records per-tick ticks via `arxana-vsatarcs-trace-build-record`; consent decisions can be a new `:writer-event-id` channel on the same stream.
   - futon1a XTDB (HTTP :7071) already holds the durable hyperedge substrate the apparatus reads via `arxana-vsatarcs-xtdb-clicks-stream-types`.

   **What's missing:** a defined stream-type for `:operator-consent-decision` that the outer loop can query, plus the bridge that pushes consent-gate dispatches into it. Likely a 50-line module + 1 stream-type registration.

2. **An outer-loop scheduler that runs at slower cadence than the inner loop.**
   The WM-side cron (M-war-machine-aif-completion R10) is already the scheduling primitive — the outer loop is a separate cron entry that runs e.g. daily instead of per-tick.
   Today there is no outer-loop entry point. **What's missing:** one `:wm-hyperparameter-inference` alias in `~/code/futon2/scripts/wm_scheduled_run.clj` (or a sibling `wm_outer_loop.clj`) that reads the operator-decision stream and updates the substrate-2 file `war-machine-terminal-vocabulary.edn` (or a v2 sibling) with learned per-class credit values.

3. **A hyperparameter-update rule** — the actual *inference*. For the smallest honest take-up: per-action-class `:intrinsic-value` is updated by a Beta(α+confirmed, β+rejected+ignored) posterior, treating each consent decision as a Bernoulli observation. The static `pragmatic-weights` become Beta posterior modes — a simple, explicit, reversible inference whose Bayesian shape is auditable. Action classes that accumulate no decisions hold prior mass — no spurious updates.

4. **A reversal hook** — Q6's "reversal protocol" specialised: the operator can emit a `:correction` evidence entry that the outer loop treats as ground-truth, overriding the inferred Beta posterior. Implementable as a special stream-type the outer loop reads with higher precision weight.

**Out of scope for the R12-unblocking handoff** (these belong to Q6's full apparatus but are NOT needed for R12 to land):
- The kill-switch ("halt the autonomous loop with one command"). The outer loop is *slower*; if it learns badly the operator has hours/days to notice before harm compounds. The kill-switch is needed before Phase-C autonomy (M-the-futon-stack:117), not before R12.
- The full debug surface ("read the EFE computation, the precision values..." at M-the-futon-stack:105). R12 needs ONE surface: the read-side queue / report that shows current per-class `:intrinsic-value` and how it changed since last outer-loop run. The full EFE/precision-surfaces debug surface is bigger and stays open.
- A general autopen mechanism. The consent-gate Joe already has (`arxana-vsatarcs-consent.el`) is sufficient input substrate for R12; promotion to autopen lives in Phase-B/C work, not here.
